API(Application Programming Interface)는 서로 다른 소프트웨어 프로그램들이 데이터를 주고받고 기능을 공유할 수 있도록 정의된 규칙과 프로토콜의 집합으로, 마치 식당의 메뉴판처럼 어떤 서비스를 요청하고 받을지를 알려주는 중개자 역할을 합니다. API 덕분에 개발자는 복잡한 내부 로직을 몰라도 다른 프로그램의 기능(예: 지도, 로그인, 결제)을 쉽게 활용하여 새로운 서비스를 만들 수 있으며, 이는 현대 소프트웨어 개발의 필수 요소입니다. 

웹 API (Web API): HTTP 프로토콜을 이용해 인터넷을 통해 통신하는 API로, 가장 흔하게 접하는 API입니다.
오픈 API (Public API): 누구나 사용할 수 있도록 공개된 API로, 공공기관이나 기업이 제공합니다.
프라이빗 API (Private API): 특정 회사나 팀 내부에서만 사용하는 API입니다. 

작동 방식 (예시: 날씨 앱)
사용자가 앱에서 '서울 날씨'를 요청합니다.
앱은 날씨 서비스 제공자의 API에 '서울'이라는 요청을 보냅니다 (API 호출).
날씨 서비스 서버는 API 규칙에 따라 요청을 처리하고, 
날씨 데이터를 JSON과 같은 형식으로 만들어 앱에 전달합니다 (API 응답).
앱은 받은 데이터를 사용자에게 보기 좋게 보여줍니다.

JSON(JavaScript Object Notation)은 이름 그대로 자바스크립트 객체 표기법에서 시작된 **'데이터를 주고받기 위한 가벼운 텍스트 형식'**입니다. 현재는 언어에 종속되지 않고 대부분의 프로그래밍 언어에서 표준처럼 사용되고 있죠.

. JSON이란 무엇인가?
JSON은 데이터를 **Key(키)**와 **Value(값)**의 쌍으로 이루어진 구조로 표현합니다. 사람이 읽기에도 편하고, 기계가 파싱(분석)하기에도 매우 효율적입니다.

기본 구조: { "이름": "값" }
지원 타입: 문자열, 숫자, 불리언(true/fales), 배열([]), 객체({}), null.

2. JSON의 등장 배경 (왜 생겨났을까?)
JSON이 등장하기 전, 웹에서 데이터를 주고받는 표준은 XML(Extensible Markup Language)이었습니다. 하지만 XML은 몇 가지 치명적인 단점이 있었습니다.

⚠️ XML의 한계
너무 무겁다 (Verbosity): 데이터를 태그(<tag></tag>)로 감싸야 하기 때문에 실제 데이터보다 태그의 크기가 더 커지는 경우가 많았습니다. 이는 네트워크 대역폭 낭비로 이어졌죠.
복잡한 파싱: 데이터를 프로그램에서 사용하려면 복잡한 DOM 파서가 필요했고, 속도도 느렸습니다.
데이터 타입의 부재: 모든 데이터가 기본적으로 텍스트로 취급되어, 숫자나 불리언을 구분하기 번거로웠습니다.

💡 더 가벼운 것이 필요해! (더글라스 크락포드의 제안)
2000년대 초반, 웹 개발자 **더글라스 크락포드(Douglas Crockford)**는 자바스크립트의 객체 선언 방식이 매우 직관적이고 가볍다는 점에 주목했습니다.
"굳이 무거운 XML 태그를 쓸 필요가 있을까? 자바스크립트 객체 표기법 그대로 문자열을 만들면 웹 브라우저에서 훨씬 빠르게 해석할 수 있잖아?"
이 아이디어가 발전하여 오늘날의 JSON 표준이 되었습니다.






1. 자바 JSON 파싱 시 5가지 주의사항
① JSON 키와 자바 변수명은 '완벽하게' 일치해야 함
자바는 대소문자를 구분합니다. JSON의 키가 userName인데 자바 변수를 username으로 만들면 데이터를 찾지 못하고 null이 들어갑니다.

해결: 스펠링과 대소문자를 두 번 확인하세요!

② 기본 생성자(Default Constructor)가 반드시 필요함
Jackson 라이브러리는 클래스를 만들 때 일단 '빈 객체'를 먼저 생성한 뒤 데이터를 채워 넣습니다. 매개변수가 있는 생성자만 만들고 빈 생성자를 빼먹으면 에러가 납니다.

해결: public MyClass() {}를 습관적으로 넣어주세요.

③ 모든 필드를 다 만들 필요는 없음 (JsonIgnore)
JSON 데이터가 100줄인데 내가 필요한 건 2줄뿐이라면? 100개의 변수를 다 만들지 않아도 됩니다.

해결: 클래스 상단에 @JsonIgnoreProperties(ignoreUnknown = true)를 붙이면, 클래스에 정의되지 않은 나머지 데이터들은 똑똑하게 무시하고 지나갑니다.

④ 숫자 vs 문자열 구분
JSON에서 "100"은 문자열이고 100은 숫자입니다. 자바 변수 타입을 int로 할지 String으로 할지 JSON의 따옴표 유무를 보고 결정해야 합니다.

⑤ NullPointerException (NPE) 주의
JSON에 특정 값이 없거나 리스트가 비어있을 경우, 자바에서는 null이 됩니다. null.something()을 호출하는 순간 프로그램이 멈춥니다.

해결: 데이터를 꺼내기 전 항상 if (data != null) 체크를 하는 습관이 중요합니다.

json 파싱하기

JSON Object와 JSON Array의 타입을 반드시 구분하자.

JSON Object (JSON 객체):

JSON 객체는 { } 로 둘러싸인 키-값 쌍의 집합.
키는 항상 문자열이고, 값은 문자열, 숫자, 객체, 배열, 불리언, 또는 null일 수 있다.
{
  "name": "홍길동",
  "age": 21,
  "subjects": ["수학", "물리", "컴퓨터 과학"]
}
 

JSON Array (JSON 배열):

JSON 배열은 [ ]로 둘러싸인 값의 순서 있는 목록.
배열의 각 값은 모든 JSON 데이터 타입이 될 수 있다.
[
  {
    "name": "홍길동",
    "age": 21,
    "subjects": ["수학", "물리", "컴퓨터 과학"]
  },
  {
    "name": "이순신",
    "age": 22,
    "subjects": ["역사", "군사학"]
  }
]
문자열(String)

숫자(Number)

불리언(Boolean)

객체(Object)

배열(List)

널(Null)
